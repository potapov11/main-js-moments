# Что происходит, когда пользователь кликает и браузер открывает другую страницу?

Ниже — путь от `click` на странице **A** до отрисовки страницы **B**: ввод, JavaScript, политики безопасности, сеть, парсинг HTML и отрисовка (**Critical Rendering Path**, CRP).

**MPA** — многостраничное приложение (каждый переход = новый HTML-документ). **SPA** — одна страница в вкладке, маршрут меняется через JS (`history.pushState`). По умолчанию разбираем **классический MPA-переход**.

---

## 0. Ссылка и кнопка — разное поведение

```html
<a id="link" href="/next">Перейти</a> <button id="btn" type="button">Сохранить</button>
```

```javascript
document.getElementById("link").addEventListener("click", (e) => {
  // e.preventDefault() — отменит переход по href
});

document.getElementById("btn").addEventListener("click", () => {
  // только ваш код; полной навигации нет, пока сами не вызовете
  // location.assign(...) или form.submit()
});
```

- **Ссылка с `href`:** браузер по умолчанию грузит новый документ (если не вызвали `preventDefault()`).
- **Кнопка `type="button"`:** навигации нет, пока вы сами её не запустите.
- **SPA:** роутер часто делает `preventDefault()` + `pushState` — полный запрос HTML не нужен. Сравнение в [разделе 7](#7-spa-vs-mpa).

## 1. На странице A: до и во время клика

### 1.1 Какой элемент «под курсором» (hit-test)

Браузер знает координаты курсора. **Hit-test** — проверка, какой элемент геометрически лежит под точкой клика (по дереву отрисовки и боксам). Берётся самый глубокий узел — например, `<span>` внутри `<a>`, а не сразу `<body>`.

Это **target** события. Он не обязан совпадать с элементом, на котором вы повесили `addEventListener`: обработчики ищутся по цепочке от target вверх к `window`.

### 1.2 Наведение: `mouseover` и `mouseenter`

| Событие      | Всплывает? | Когда срабатывает                                                         |
| ------------ | ---------- | ------------------------------------------------------------------------- |
| `mouseover`  | Да         | При входе на элемент и при переходе на вложенных детей — снова на предках |
| `mouseenter` | Нет        | Один раз при входе в границы элемента, без повторов на детях              |

Упрощённый порядок при наведении на вложенный узел:

```
mouseenter (на предках)
  → mouseover (от target вверх, всплытие)
```

### 1.3 Фазы события и `click`

Любое событие с всплытием идёт в три фазы:

1. **Capture** — от `window` к target
2. **Target** — на элементе клика
3. **Bubble** — от target к `window`

```javascript
parent.addEventListener("click", () => console.log("bubble"), false);
parent.addEventListener("click", () => console.log("capture"), true);
child.addEventListener("click", () => console.log("target"));
// клик по child: capture → target → bubble
```

`click` появляется после `pointerdown` → `pointerup` (или `mousedown`/`mouseup`) **на том же** target (если палец/курсор почти не сдвинулись).

| Свойство              | Значение                                                               |
| --------------------- | ---------------------------------------------------------------------- |
| `button`              | `0` — левая, `1` — средняя (колёсико), `2` — правая                    |
| `clientX` / `clientY` | Координаты относительно видимой области окна (viewport)                |
| `target`              | Элемент из hit-test                                                    |
| `currentTarget`       | Элемент, на котором сейчас выполняется обработчик                      |
| `bubbles`             | у `click` — `true`                                                     |
| `cancelable`          | `true` — можно отменить действие по умолчанию через `preventDefault()` |

```javascript
link.addEventListener("click", (e) => {
  // false, если кликнули по <span> внутри ссылки
  console.log(e.target === link);
  // всегда link — на нём висит listener
  console.log(e.currentTarget === link);
});
```

**Другие клики по ссылке:** Ctrl+клик / Cmd+клик — часто новая вкладка; средняя кнопка — отдельная ветка. Это не всегда тот же код, что обычный левый клик.

### 1.4 Event Loop и макротаски при клике

На вкладку обычно приходится **один главный поток**: в нём JS, часть layout и парсинга. Пока поток занят долгим синхронным кодом, UI не отвечает — новые события **ждут в очереди**.

**Важно:** `addEventListener` **сам по себе не макротаска**. Колбэк тоже не макротаска — это просто функция. **Макротаска** — задача «обработать событие» (клик, таймер, ответ сети), которую Event Loop достаёт из очереди; **внутри** этой задачи браузер **синхронно** вызывает ваши обработчики.

| Сущность | Макротаска? |
|----------|-------------|
| `addEventListener('click', fn)` — вызов регистрации | Нет, синхронный API |
| Функция `fn`, переданная в listener | Нет, это колбэк |
| Задача «обработать click» после физического клика | **Да** |
| Выполнение `fn()` при обработке события | Нет, синхронная часть макротаски |

**Три этапа**

1. **Регистрация** (синхронно, сразу при выполнении скрипта):

```javascript
button.addEventListener('click', () => console.log('клик'));
console.log('готово'); // выполнится до любого клика
// браузер только запоминает обработчик, очередь не трогает
```

2. **Клик пользователя** — hit-test, объект события, в очередь макротасок попадает задача вроде «вызвать обработчики `click` на цепочке элементов».

3. **Event Loop** берёт эту макротаску и **синхронно** гоняет capture → target → bubble; на каждом узле вызывает `fn()`. После макротаски — microtasks, затем следующая макротаска.

```
Синхронный код:  addEventListener(...) → console.log('готово')
Клик:            [очередь макротасок: обработать click]
Event Loop:      макротаска click → синхронно все fn() → конец макротаски
```

**Проверка порядка** (после клика по кнопке):

```javascript
console.log('1. начало');

button.addEventListener('click', () => {
  console.log('3. колбэк клика');
  setTimeout(() => console.log('5. setTimeout — новая макротаска'), 0);
  Promise.resolve().then(() => console.log('4. Promise — microtask'));
});

console.log('2. конец синхронного кода');
// 1 → 2 → (клик) → 3 → 4 → 5
```

Microtask (`4`) выполняется **до выхода** из обработки клика, но **после** синхронного кода внутри колбэка. `setTimeout` (`5`) — уже **следующая** макротаска.

**Два listener на одном элементе** (один клик — одна макротаска):

```javascript
document.body.addEventListener('click', () => {
  console.log('A');
  setTimeout(() => console.log('B'), 0);
  Promise.resolve().then(() => console.log('C'));
});
document.body.addEventListener('click', () => console.log('D'));
console.log('E');
// после клика: E → A → C → D → B
```

**Ещё нюансы**

- События от пользователя (клик, мышь, клавиатура) — через очередь **макротасок**.
- `dispatchEvent()` из JS — **синхронно**, без новой макротаски (обработчики сразу на текущем стеке).
- Долгий синхронный цикл (`while (true)`) — клик **встанет в очередь** и выполнится после; иногда события **объединяют** (coalescing).

Очереди в целом:

```
Call Stack           — синхронный JS сейчас
Microtasks           — Promise.then, queueMicrotask, MutationObserver
Macrotasks           — click, setTimeout, I/O, …
requestAnimationFrame — перед отрисовкой кадра
```

После обработчиков `click` (и microtasks внутри этой макротаски), если не вызвали `preventDefault()`, браузер выполняет **переход по ссылке** — это уже следующие шаги навигации.

## 2. Обработчики и переход по `href`

### 2.1 Когда браузер решает «уйти на другую страницу»

1. Hit-test → target.
2. По правилам HTML для `<a href="...">`: если URL валидный и событие не отменено — **перейти по ссылке**.
3. `<button type="submit">` в форме — отправка формы (другой сценарий навигации).
4. `href="#"` — прокрутка; `javascript:...` — выполнение скрипта, не полная загрузка страницы.

Сначала выполняются ваши listeners, потом (если не отменили) — **действие по умолчанию**.

### 2.2 Порядок: listeners → microtasks → переход

См. [§ 1.4](#14-event-loop-и-макротаски-при-клике): макротаска — это **обработка события**, а не `addEventListener` и не колбэк.

На ссылке несколько `click`-listener'ов отработают **в одной** макротаске (порядок регистрации + фазы capture/bubble). **Microtasks** из обработчиков выполнятся до её завершения. **Переход по `href`** — после обработчиков, если не было `preventDefault()`.

### 2.3 `preventDefault()` и `stopPropagation()`

```javascript
link.addEventListener("click", (e) => {
  e.preventDefault(); // событие cancelable и навигация ещё не зафиксирована
});
```

Браузер помечает событие как отменённое — **переход по этой ссылке для этого клика не начинается**, HTTP-запрос за HTML не уходит.

Если навигацию уже запустили вручную (`location.href = '...'`), это отдельный путь.

`stopPropagation()` **не отменяет** переход по `<a>` на предке: действие по умолчанию смотрит на **target**. Но обработчики выше по дереву могут не вызваться, если остановили всплытие на дочернем элементе.

### 2.4 Несколько обработчиков

- На одном элементе — в порядке регистрации.
- На цепочке — capture → target → bubble.
- `once: true` — снимается после первого срабатывания.

## 3. Переход начался — страница A выгружается

### 3.1 Проверки перед уходом

Перед загрузкой страницы B браузер проверяет политики страницы A и ответа сервера.

#### CSP (Content Security Policy)

**CSP** — политика безопасности от сервера: белый список того, что страница может загружать и куда может вести пользователя. Задаётся заголовком ответа (предпочтительно) или мета-тегом:

```http
Content-Security-Policy: default-src 'self'; script-src 'self' https://cdn.example.com
```

```html
<meta http-equiv="Content-Security-Policy" content="default-src 'self'">
```

Браузер сверяет каждый ресурс и действие с **директивами**. Нарушение → блокировка, запись в консоль (CSP violation). Цель — ограничить ущерб от XSS: даже внедрённый `<script>` может не выполниться, если его источник не в `script-src`.

**При клике по ссылке** проверяется, разрешён ли переход на `href` с текущей страницы A:

| Директива | Что ограничивает |
|-----------|------------------|
| **`navigate-to`** | Переходы по ссылке, `location.assign`, часть сценариев `window.open` |
| **`form-action`** | URL, куда уходит `<form method="get\|post">` |

Пример:

```http
Content-Security-Policy: navigate-to 'self' https://partner.example.com
```

Переход на `https://evil.com` — навигация **блокируется**. Поддержка `navigate-to` в браузерах не везде одинаковая; на собеседовании важна идея: CSP режет не только скрипты, но и **куда можно увести вкладку**.

Другие частые директивы (для контекста, не только навигация):

| Директива | Смысл |
|-----------|--------|
| `default-src` | Значение по умолчанию для остальных |
| `script-src` | Откуда можно выполнять JS |
| `style-src` | CSS |
| `connect-src` | `fetch`, XHR, WebSocket |
| `img-src` | Изображения |
| `frame-src` | `iframe` |
| `base-uri` | Допустимый `<base href>` |

#### Остальные проверки

- **Sandbox** у `<iframe>` — не CSP: атрибут `sandbox` ограничивает встроенную страницу (в т.ч. выход в top-level, если нет `allow-top-navigation`).
- **Mixed content** — HTTP-ресурсы на HTTPS-странице; браузер блокирует или режет по правилам.
- **Permissions Policy** (бывш. Feature Policy) — можно ли на B камера, геолокация, autoplay и т.д.
- **Content-Disposition: attachment** — сервер просит **скачать** файл, а не открыть как HTML-документ во вкладке.

### 3.2 `beforeunload`

Если есть обработчик, браузер может показать «Уйти со страницы?». Свой текст в диалоге в современных браузерах **нельзя** — только стандартное сообщение. `await` внутри **не надёжно** блокирует уход.

```javascript
window.addEventListener("beforeunload", (e) => {
  e.preventDefault();
  e.returnValue = "";
});
```

### 3.3 Состояния вкладки: Hidden, Frozen

| Состояние      | Смысл                                                                                              |
| -------------- | -------------------------------------------------------------------------------------------------- |
| **Active**     | Вкладка на экране, JS работает                                                                     |
| **Passive**    | Видна, с ограничениями (например, скролл)                                                          |
| **Hidden**     | Ушли с вкладки или началась навигация — `visibilityState === 'hidden'`, событие `visibilitychange` |
| **Frozen**     | JS **приостановлен**: таймеры и `requestAnimationFrame` не идут                                    |
| **Discarded**  | Вкладку выгрузили из памяти (не всегда)                                                            |
| **Terminated** | Документ уничтожен                                                                                 |

При уходе: **Hidden** → пауза таймеров и rAF → событие **`pagehide`**. У `pagehide` есть `event.persisted` — страница попала в **bfcache** (кэш «Назад/Вперёд»), см. ниже.

### 3.4 JS на A останавливается

Долгие задачи обрываются вместе с документом.

**Частая ошибка:** `setInterval`, открытый `WebSocket`, `fetch` без отмены — мешают сборке мусора и bfcache.

```javascript
const id = setInterval(() => {
  console.log(window.document.title); // замыкание держит document
}, 1000);
// clearInterval(id) в pagehide или visibilitychange
```

## 4. Сеть: запрос новой страницы

Запрос навигации (**navigation request**) в Chromium часто важнее, чем загрузка картинок.

### 4.1 От клика до HTTP

После обработчиков `click` браузер знает **куда** идти. Дальше сетевой стек превращает `href` в соединение с сервером и только потом отправляет `GET` за HTML. Часть шагов может **пропуститься**, если домен уже «тёплый» (кэш DNS, открытое соединение, TLS session ticket).

#### Шаг 1: полный URL

Относительный путь из ссылки склеивается с **base URL** текущего документа:

```
Страница A: https://example.com/blog/post-1
Ссылка:     <a href="/next">
Итог:       https://example.com/next
```

Проверяются схема (`https`), хост, порт, путь. Для другого origin — отдельное соединение и другие правила cookies (`SameSite`, third-party).

#### Шаг 2: HSTS (только HTTPS)

**HSTS** (HTTP Strict Transport Security) — браузер **запоминает**: этот хост нужно открывать только по HTTPS.

- Первый визит: сервер прислал заголовок `Strict-Transport-Security: max-age=...`
- Список **HSTS preload** в Chrome/Firefox — домен уже HTTPS **до** первого запроса
- Если в ссылке был `http://`, браузер **внутренне** подменит на `https://` (или заблокирует downgrade)

#### Шаг 3: DNS — имя → IP

Браузеру нужен IP сервера. Поиск идёт слоями, пока не найдёт ответ:

| Уровень | Что это |
|---------|---------|
| Кэш браузера | Недавно уже резолвили `example.com` |
| Кэш ОС | Системный кэш DNS |
| DoH / рекурсивный резолвер | Запрос к DNS-провайдеру (1.1.1.1, роутер, ISP) |
| Авторитативные серверы | Цепочка от корня `.` до записи `A` / `AAAA` |

Итог: `example.com` → `93.184.216.34` (IPv4) или IPv6-адрес.

#### Шаг 4: TCP — канал до сервера

**TCP** — надёжное соединение «клиент ↔ сервер» (порт обычно **443** для HTTPS).

- **Первый раз:** три этапа рукопожатия (SYN → SYN-ACK → ACK), затем данные
- **Повторный визит:** может переиспользоваться **keep-alive** / пул соединений — быстрее

#### Шаг 5: TLS (шифрование для HTTPS)

Поверх TCP — **TLS**. Упрощённо:

1. Клиент: «поддерживаю такие версии и шифры» (ClientHello)
2. Сервер: **сертификат** + выбранный шифр (ServerHello)
3. Браузер **проверяет сертификат** (см. ниже) — без успеха соединение обрывается (`NET::ERR_CERT_*`)
4. Общий секрет → дальше HTTP **внутри** шифра

**Session resumption** — если уже были на этом хосте, рукопожатие короче (ticket / 0-RTT в части сценариев).

#### Сертификат, CA и цепочка доверия

**TLS-сертификат** — электронный документ, который говорит: «этот **публичный ключ** принадлежит хосту `example.com`». Сервер доказывает владение **приватным ключом** (пара к публичному) во время рукопожатия TLS.

**CA (Certificate Authority, центр сертификации)** — организация, которой браузеры и ОС **заранее доверяют**. CA своей подписью подтверждает: «мы проверили владельца домена, этот сертификат настоящий». Примеры: Let's Encrypt, DigiCert, GlobalSign.

Цепочка обычно из трёх уровней:

```
[Сертификат сервера]     example.com  ← отдаёт сайт при TLS
        ↑ подписан
[Промежуточный CA]       R3 / WE1 …   ← часто 1–2 звена
        ↑ подписан
[Корневой CA]            ISRG Root …  ← лежит в хранилище доверия браузера/ОС
```

**Корневые CA** встроены в Windows, macOS, Android, Chrome — это **trust store** (хранилище доверенных корней). Браузер поднимается от сертификата сайта вверх и на каждом звене проверяет **цифровую подпись** предыдущего уровня. Если дошли до корня из своего списка — цепочка **доверенная**.

**Что именно проверяет браузер**

| Проверка | Суть |
|----------|------|
| **Цепочка до доверенного CA** | Все подписи валидны, корень в trust store |
| **Срок действия** | `notBefore` / `notAfter` — сертификат не просрочен |
| **Имя хоста** | В URL `https://example.com`, а в сертификате в **SAN** (Subject Alternative Name) или CN есть `example.com` (или `*.example.com` по правилам wildcard) |
| **Отзыв (опционально)** | OCSP / CRL — сертификат не отозван (например, после утечки ключа) |
| **Политики** | Certificate Transparency, иногда pinning (редко в обычных сайтах) |

Если что-то не сходится — страница **не откроется** (красный экран «Ваше подключение не защищено»), кроме случая, когда пользователь вручную добавил исключение (не делайте так в проде).

**Самоподписанный сертификат** — подписан не CA из trust store, а самим собой. Браузер **не доверяет** по умолчанию (типично для `localhost` в разработке).

**Зачем это при клике по ссылке:** вы переходите на `https://…` — браузер должен убедиться, что говорите с **настоящим** сервером `example.com`, а не с подменой в сети (MITM). После проверки трафик (включая cookies и HTML) идёт **зашифрованным**.

```
Сервер отдал cert для example.com
       │
       ▼
Подпись совпадает с Intermediate CA? ──нет──► ERR_CERT_AUTHORITY_INVALID
       │ да
       ▼
Корень в trust store? ──нет──► ошибка доверия
       │ да
       ▼
Срок OK? Имя хоста = URL? ──нет──► ERR_CERT_DATE_INVALID / COMMON_NAME_INVALID
       │ да
       ▼
TLS установлен → HTTP GET /next
```

#### Шаг 6: ALPN — какой HTTP поверх TLS

В TLS клиент и сервер договариваются через **ALPN**:

| Протокол | Транспорт | Кратко |
|----------|-----------|--------|
| **HTTP/1.1** | TCP + TLS | Один запрос за раз на соединении (без мультиплексирования) |
| **HTTP/2** | TCP + TLS | Много запросов в одном соединении, сжатие заголовков |
| **HTTP/3** | **QUIC** (UDP) | Потоки без блокировки друг друга при потере пакета; быстрее восстановление |

**HTTP/3** не «HTTP поверх обычного TCP»: QUIC сам даёт шифрование и мультиплексирование на UDP — меньше **head-of-line blocking**, чем у HTTP/2 при плохой сети.

#### Схема: браузер и сервер

```
  БРАУЗЕР (вкладка)                         СЕРВЕР
  ─────────────────                         ──────

  href="/next"
       │
       ▼
  URL: https://example.com/next
       │
       ▼
  HSTS: только HTTPS ✓
       │
       ▼
  DNS ──────────────────────────────────►  (запись A у DNS-провайдера)
       ◄──────────────────────────────────  IP: 93.184.216.34
       │
       ▼
  TCP :443  ─── SYN / SYN-ACK / ACK ─────►  слушает 443
       │      (или reuse старого сокета)
       ▼
  TLS  ─── ClientHello / ServerHello ────►
       │    сертификат, проверка CA
       │    общий ключ шифрования
       ▼
  ALPN: "h2" или "h3" или "http/1.1"
       │
       ▼
  HTTP GET /next  ───────────────────────►
       │    Host, Cookie, Referer,
       │    Sec-Fetch-Mode: navigate
       ◄──────────────────────────────────
       │    200 OK + тело HTML (потоком)
       ▼
  Парсер HTML → DOM (§ 5)
```

#### Схема: порядок во времени

```
click обработан → навигация зафиксирована
    │
    ├─ [может быть 0 мс]  URL + HSTS + DNS из кэша
    │
    ├─ [десятки–сотни мс] DNS запрос (если нет в кэше)
    │
    ├─ [RTT]              TCP + TLS (дольше на холодном старте)
    │
    ├─ [RTT]              HTTP GET /next
    │
    └─► байты HTML ──────► парсер (не ждёт конца файла)
```

**RTT** (round-trip time) — время «туда-обратно» до сервера; на мобильной сети DNS + TLS + первый байт HTML часто заметнее, чем размер HTML.

Повторный переход на **тот же origin** обычно быстрее: DNS и TLS уже в кэше, соединение может быть открыто.

### 4.2 Заголовки запроса

```
GET /next HTTP/2
Host: example.com
Referer: https://example.com/page-a
Cookie: session=...
Sec-Fetch-Dest: document
Sec-Fetch-Mode: navigate
Sec-Fetch-Site: same-origin | cross-site
Sec-Fetch-User: ?1
```

`Sec-Fetch-*` показывает: это переход документа, а не `fetch()` из скрипта и не картинка.

### 4.3 Ответ, редиректы, потоковый HTML

| Код                     | Что делает браузер                                         |
| ----------------------- | ---------------------------------------------------------- |
| **200**                 | Парсит HTML из тела ответа                                 |
| **301/302/303/307/308** | Идёт по новому URL (лимит ~20 редиректов)                  |
| **304**                 | Чаще для отдельных файлов; для навигации свои правила кэша |
| **404**                 | Страница ошибки или HTML с 404                             |

HTML **не ждут целиком**: байты сразу идут в потоковый парсер → DOM растёт по мере загрузки → раньше стартуют запросы картинок и скриптов.

### 4.4 Service Worker

Если URL в scope воркера:

```
запрос навигации → fetch в Service Worker → кэш / сеть / свой Response → парсер HTML
```

Service Worker живёт **отдельно** от документа A и может отдать B из кэша без сети.

## 5. Страница B: от HTML до экрана (CRP)

### 5.1 Новый документ

Вкладка получает новый **Document** (кратко может мелькнуть `about:blank`). Старый `document` A отвязывается.

**bfcache:** при «Назад» A иногда восстанавливается из памяти без полной перезагрузки — на `pageshow` будет `event.persisted === true`.

### 5.2 Парсинг HTML и скрипты

```html
<link rel="stylesheet" href="/app.css" />
<!-- блокирует первую отрисовку -->
<script src="/blocking.js"></script>
<!-- блокирует парсинг DOM -->
<script src="/defer.js" defer></script>
<!-- после DOM, порядок файлов сохранён -->
<script src="/async.js" async></script>
<!-- как загрузится; порядок не гарантирован -->
<script type="module" src="/m.js"></script>
<!-- как defer + strict mode -->
```

Обычный `<script>` без `async`/`defer`/`module`: парсер **останавливается**, выполняет JS, потом продолжает DOM.

### 5.3 Стили и Render Tree

CSS **не останавливает** построение DOM, но без **CSSOM** браузер обычно **не рисует** первый кадр (чтобы не было «голого» HTML). DOM + CSSOM → **Render Tree** (без `display: none`).

### 5.4 Layout → Paint → Composite

```
Render Tree → Layout (размеры и позиции) → Paint (пиксели) → Composite (слои, GPU)
```

**Layout thrashing** — в цикле читать `offsetHeight` и сразу менять стили: лишние пересчёты layout.

### 5.5 Когда страницу можно считать готовой

| Метрика / событие    | Когда                                                                                 |
| -------------------- | ------------------------------------------------------------------------------------- |
| **FCP**              | Первый текст или картинка на экране                                                   |
| **DOMContentLoaded** | HTML разобран, `defer` / `module` выполнены; картинки из разметки ещё могут грузиться |
| **load**             | Загружены ресурсы из разметки (img, css в тегах, iframe)                              |
| **LCP**              | Самый крупный видимый блок (Core Web Vitals)                                          |
| **TTI**              | DOM готов и нет долгих блокирующих задач — страница отзывчива                         |

После навигации снова **Active**, `visibilitychange` → `visible`. Ссылка с `#section` — прокрутка к якорю.

```javascript
document.addEventListener("DOMContentLoaded", () => {
  console.log("DOM готов");
});
window.addEventListener("load", () => {
  console.log("Ресурсы из разметки загружены");
});
```

## 7. SPA vs MPA

```javascript
document.addEventListener("click", (e) => {
  const a = e.target.closest('a[href^="/"]');
  if (!a || e.metaKey) return;
  e.preventDefault();
  history.pushState({}, "", a.pathname);
  renderRoute(a.pathname);
});
```

|              | **MPA**           | **SPA**                     |
| ------------ | ----------------- | --------------------------- |
| Документ     | Новый каждый раз  | Тот же                      |
| Сеть         | HTML целиком      | API + JS-chunks             |
| CRP          | Полный            | Частичное обновление DOM    |
| `load` / DCL | На каждый переход | В основном при первом входе |
| SEO          | HTML с сервера    | Нужен SSR/SSG или пререндер |

## 8. Схема: от `click` до `load`

```
[Страница A]
  hit-test → mouseenter/mouseover → pointerdown/up → click (capture → target → bubble)
       │
       ├─ preventDefault? → ДА → остаёмся на A
       │
       НЕТ
       ├─ beforeunload?
       ├─ visibilitychange: hidden
       ├─ pagehide
       ├─ freeze JS на A
       │
[Сеть]
  DNS → TCP → TLS → GET (Sec-Fetch-Mode: navigate)
       ├─ редиректы
       └─ 200, поток HTML
       │
[Страница B]
  DOM → CSSOM → Render Tree → Layout → Paint → Composite
  FCP → DOMContentLoaded → visible → load
```

## 9. Вопросы на собеседовании

**Почему клик запаздывает, если main thread занят 5 секунд?**  
Событие ждёт в очереди, пока не закончится синхронный код.

**`addEventListener` — это макротаска?**  
Нет. Регистрация — синхронный вызов. Макротаска — задача «обработать click»; колбэк вызывается **синхронно** внутри неё.

**`preventDefault` на `click` по ссылке — хватит, чтобы не уйти?**  
Да, для перехода по этому клику. Нет, если уже вызвали `location.assign` или submit формы.

**`DOMContentLoaded` и `load`?**  
DCL — DOM и отложенные скрипты готовы. `load` — ещё подгрузились ресурсы из разметки.

**CSS блокирует парсинг HTML?**  
Нет. Блокирует **первую отрисовку**, пока не построен CSSOM.

**`Sec-Fetch-Mode: navigate`?**  
Признак запроса из перехода по странице, не из `fetch()` в скрипте.

**Что такое CSP?**  
Политика от сервера (заголовок `Content-Security-Policy`): что можно грузить на страницу и куда можно навигировать (`navigate-to`, `form-action`).

**HSTS?**  
Браузер помнит: этот домен только по HTTPS.

**Что такое CA при HTTPS?**  
Центр сертификации, которому доверяют браузеры; он подписывает сертификат сервера. Браузер строит цепочку до корневого CA из trust store и проверяет подпись, срок и имя хоста.

**Critical Rendering Path?**  
DOM + CSSOM → Render Tree → Layout → Paint → Composite.

**Таймеры на старой странице?**  
В Frozen не тикают, но замыкания на `document` мешают памяти и bfcache.

**Service Worker при клике по ссылке?**  
Может отдать HTML из кэша без сети.

**MPA vs SPA на `<a href>`?**  
MPA — новый документ и полный CRP. SPA — `preventDefault` + `pushState`, если роутер перехватил.

## 10. Отладка в Chrome

1. **Performance → Event Log** — `click`, ваш код, navigation.
2. **Network** — тип `document`, редиректы, кэш.
3. **Application → Back/forward cache** — почему A не в bfcache.
4. **Performance / Coverage** — блокирующие скрипты и CSS.
5. В коде — снимать таймеры и подписки в `pagehide` / `visibilitychange`.

Переход по ссылке — не один вызов функции, а цепочка: ввод → Event Loop → правила HTML → сеть → парсинг → отрисовка → жизненный цикл вкладки. Это нужно понимать при MPA, SPA, bfcache и метриках загрузки.
