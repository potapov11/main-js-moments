# Fiber — архитектура reconciler’а

Fiber появился, чтобы reconciler мог **останавливать, возобновлять и приоритизировать** работу. Старый stack reconciler шёл рекурсивно и не умел отдавать управление браузеру mid-render.

Связано: [Virtual DOM](virtual-dom.md), [рендеринг](rendering.md).

---

## Ключевые идеи

- **Fiber** — внутренний объект (узел), соответствующий экземпляру компонента / DOM-элементу / фрагменту.
- Дерево Fiber — **связный список** (`child` / `sibling` / `return`), а не только дерево указателей «в глубину».
- Работа делится на **units of work**: можно прервать обход, обработать события / отрисовать кадр, продолжить.
- Есть **current** дерево (на экране) и **workInProgress** (черновик следующего кадра).

---

## 1. Проблема до Fiber

Рекурсивный обход большого дерева:

1. Начал reconciliation — нельзя «вежливо» остановиться.
2. Долгий sync-обход блокирует main thread → фризы UI, delayed input.
3. Нет нормальных **приоритетов**: ответ на клик не может обогнать тяжёлый рендер списка.

Fiber = переход к **инкрементальному** reconciler’у.

---

## 2. Что хранит Fiber-узел (упрощённо)

Каждый Fiber roughly знает:

| Поле / идея | Зачем |
|-------------|--------|
| `type`, `key` | тип компонента/тега, ключ для списков |
| `pendingProps` / `memoizedProps` | входящие и зафиксированные props |
| `memoizedState` | state (и цепочки хуков) |
| `child` / `sibling` / `return` | обход дерева как списка |
| `alternate` | связь current ↔ workInProgress |
| `flags` / effects | что сделать в commit (update, placement, deletion…) |
| `lanes` / priority | насколько срочно это обновление |

Для ответа на собесе не нужно цитировать исходники — достаточно модели «узел работы + связный список + приоритет».

---

## 3. Linked list вместо чистой рекурсии

Обход:

```
A
├── B
│   └── D
└── C
```

в Fiber-терминах: `A.child → B`, `B.sibling → C`, `B.child → D`, у всех `return` на родителя.

Так reconciler может:

1. Взять следующий unit of work.
2. Сделать кусок работы.
3. Проверить: «есть ли срочные задачи / нужно ли отдать кадр?»
4. Продолжить или **yield**.

Именно это открывает Concurrent Features (time slicing): работа *может* растягиваться на несколько кадров.

!!! note "Concurrent не значит «многопоточность»"
    JS по-прежнему на одном потоке. «Параллельность» — в смысле *чередования* работы React с браузером и приоритетами, не Web Workers внутри reconciler’а.

---

## 4. Double buffering: current и workInProgress

- **current** — дерево, соответствующее UI на экране.
- **workInProgress** — дерево, которое строится во время Render phase.
- У каждого Fiber есть `alternate` — парный узел в другом дереве.

Когда Render phase успешно завершён, в Commit phase указатель **корня** переключается: workInProgress становится новым current. Это похоже на double buffering в графике: готовим кадр «за кадром», потом показываем атомарно.

---

## 5. Приоритеты и lanes (идея)

Обновлениям назначается приоритет (в современных терминах — **lanes**):

- высокий: ввод пользователя, анимации;
- ниже: data fetch, несрочные обновления.

React может:

- начать низкоприоритетный рендер списка;
- прервать его ради срочного обновления от клика;
- позже продолжить или пересчитать с актуальным state.

Это связано с API вроде `startTransition` / `useDeferredValue`: вы *явно* помечаете обновление как несрочное.

## Типичные формулировки

| Вопрос | Короткий ответ |
|--------|----------------|
| Что такое Fiber? | Внутренняя единица работы / узел дерева reconciler’а |
| Зачем Fiber? | Прерываемый рендер, приоритеты, лучше отзывчивость UI |
| Fiber = Virtual DOM? | Fiber — реализация; VDOM — концепт описания UI |

---

## Шпаргалка

- Fiber = узел + unit of work.
- Связный список → можно останавливаться между единицами работы.
- current / workInProgress → подготовка следующего UI без рваного commit.
- Приоритеты → срочный ввод важнее фонового рендера.
