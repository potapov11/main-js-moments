# Virtual DOM и reconciliation

На собеседовании ждут не лозунг «Virtual DOM быстрее DOM», а понимание: React описывает UI как **дерево объектов**, сравнивает его с предыдущим снимком и меняет **реальный DOM минимально**.

Связано: [Fiber](fiber.md), [рендеринг](rendering.md).

---

## Ключевые идеи

- **Virtual DOM** — in-memory представление UI (элементы / Fiber-узлы), а не «особая быстрая версия DOM».
- React сравнивает **новое** дерево с **предыдущим** (reconciliation) и планирует точечные мутации.
- Diff не «магически полный за O(1)»: эвристики (тип, `key`, позиция) делают сравнение практичным.
- Реальный DOM дорогой (layout/paint); Virtual DOM помогает *решить, что менять*, а не заменяет браузерный движок.

---

## 1. Что такое Virtual DOM

Когда пишете JSX:

```jsx
const element = <Button label="OK" />;
```

после транспиляции это примерно:

```javascript
const element = React.createElement(Button, { label: 'OK' });
```

Результат — **plain object** (React element): тип, props, children. Дерево таких объектов и называют Virtual DOM в разговорном смысле.

| Слой | Что это |
|------|---------|
| JSX / `createElement` | Описание «что хотим видеть» |
| Virtual tree / Fiber tree | Внутреннее дерево, с которым работает reconciler |
| Real DOM | То, что видит браузер и пользователь |

!!! tip "Формулировка для интервью"
    Virtual DOM — представление UI в виде JS-объектов. React сверяет его с предыдущим и обновляет реальный DOM только там, где нужно.

---

## 2. Зачем он нужен

Прямая работа с DOM вручную сложна: легко забыть синхронизировать UI и данные, сложно отменять частичные обновления, дорого трогать узлы «на всякий случай».

React даёт **декларативность**: «UI = f(state)». При смене state компонент **перевызывается**, получается новое описание UI; reconciler выясняет diff.

Миф: «Virtual DOM всегда быстрее императивных обновлений». На практике руками написанный точечный DOM-апдейт может быть быстрее. Выигрыш React — **модель программирования** + достаточная скорость за счёт батчинга и умного diff.

---

## 3. Reconciliation (согласование)

**Reconciliation** — алгоритм сравнения предыдущего и нового дерева и построения плана обновлений.

Упрощённые правила (то, что обычно спрашивают):

1. **Разный тип элемента** (`div` → `span`, `Header` → `Footer`) → старое поддерево сносится, новое монтируется с нуля (state внутри теряется).
2. **Тот же тип** → обновляются props, дети сравниваются рекурсивно.
3. **Списки детей** — для стабильной идентификации нужен **`key`**.

```
prev:  <ul><li key="a">A</li><li key="b">B</li></ul>
next:  <ul><li key="b">B</li><li key="a">A</li></ul>
```

С `key` React понимает: узлы *переехали*, а не «удалили A и создали заново». Без ключей (или с `key={index}` при перестановках) — лишние unmount/mount и баги state.

### Ключи (`key`)

- Уникальны **среди siblings**, не глобально.
- Стабильны во времени (id из данных, не индекс при insert/reorder/filter).
- Не про «ускорение» сами по себе — про **идентичность** элемента между рендерами.

---

## 4. Virtual DOM ≠ Fiber

Исторически говорили «Virtual DOM». В современном React единица внутренней работы — **Fiber** (см. [fiber.md](fiber.md)): linked-list структура, которую можно обходить прерывно.

Для собеса:

- «Virtual DOM» — концепт описания UI + идея diff.
- «Fiber» — реализация reconciler’а, которая это описание обрабатывает.

---

## Типичные ошибки на собесе

| Ошибка | Лучше сказать |
|--------|----------------|
| «VDOM всегда быстрее реального DOM» | VDOM решает *что* менять; скорость — компромисс модели и эвристик |
| «React всегда обновляет весь DOM» | Обычно точечные мутации после diff |
| «`key={index}` всегда ок» | Ок для статичных списков; опасно при reorder/insert |

---

## Шпаргалка

- Virtual DOM — дерево описаний UI в памяти.
- Reconciliation — сравнение prev/next + план обновлений.
- Тип сменился → remount; тот же тип → update props/children.
- `key` = стабильный id sibling’а для списков.
