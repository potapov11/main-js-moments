# `throttle(fn, delay)` — теория и мини‑задача

`throttle` ограничивает частоту вызовов: функция выполняется **не чаще**, чем раз в `delay` мс.

Это practical‑задача на **замыкания**, **таймеры** и аккуратную работу со временем.

---

## Интуиция

Пример: событие `scroll` может приходить десятки раз в секунду.

- Без throttle — тяжёлый обработчик перегружает поток.
- С throttle — обработчик срабатывает **примерно раз в N мс**.

---

## Поведение (leading / trailing)

Часто обсуждают два варианта:

- **leading**: сработать сразу при первом событии
- **trailing**: сработать в конце окна, если события были

Минимум для старта: сделать только `leading` (или только `trailing`) и проговорить, как добавить второй режим.

---

## Базовая реализация (leading)

```javascript
function throttle(fn, delay) {
  let lastCall = 0;

  return function (...args) {
    const now = Date.now();
    if (now - lastCall < delay) return;

    lastCall = now;
    fn(...args);
  };
}
```

### Важные детали

- `lastCall` живёт в **замыкании** и хранит время последнего выполнения.
- `...args` / `fn(...args)` — rest/spread.

---

## Пример использования

```javascript
const onScroll = throttle(() => {
  console.log('scroll tick');
}, 200);

window.addEventListener('scroll', onScroll);
```

---

## Типичные ошибки

| Ошибка | Почему плохо | Как исправить |
|-------|-------------|--------------|
| Троттлить «внутри» обработчика | создаётся новый throttle на каждый вызов | обёртку создавать **один раз** |
| Перепутать debounce/throttle | поведение другое | проговорить: debounce — «после паузы», throttle — «не чаще N мс» |
| Потерять `this` | методы ломаются | `fn.apply(this, args)` |

---

## Мини‑задачи для практики

1) Реализуй `throttle(fn, delay, { leading, trailing })`.

2) Добавь `cancel()` — отменить запланированный trailing‑вызов.

---

## Шпаргалка для собеседования

- `throttle` — «вызывать **не чаще** раз в N мс».
- Минимальная реализация: `lastCall` (или таймер) в замыкании.
