## React: key и реконсилиация

### Вопрос

В списке элементов используется `key={index}` вместо `key={task.id}`. Пользователь редактирует элемент, затем список сортируется. Что произойдёт с локальным state/input и почему?

### Проблема в ответе

Ответ был близко по смыслу, но механизм был сформулирован неточно: было сказано про «потерю ссылки». На самом деле React не теряет ссылку, а ошибочно считает, что компонент на том же индексе — это та же логическая сущность.

### Правильная мысль

`key` задаёт identity элемента для реконсилиации. `key={index}` привязывает state к позиции в списке, а не к данным. При сортировке props меняются, но локальный state остаётся у прежней позиции, из-за чего input/focus могут оказаться у другой задачи.

## React: useEffect, cleanup и зависимости

### Вопрос

Зачем нужен cleanup в `useEffect` с `fetchUser(userId)`? Чем отличается `useEffect(fn)` от `useEffect(fn, [])`?

### Проблема в ответе

Сначала было сказано, что cleanup нужен, чтобы «размонтировался useEffect» и fetch не дёргался лишний раз. Также было перепутано отсутствие второго аргумента и пустой массив зависимостей.

### Правильная мысль

Cleanup убирает побочные эффекты прошлого запуска: отменяет запрос, чистит подписки, таймеры, event listener'ы. При `[userId]` cleanup вызывается перед следующим запуском эффекта и при unmount.

- `useEffect(fn)` — после каждого рендера.
- `useEffect(fn, [])` — при mount, cleanup при unmount.
- `useEffect(fn, [userId])` — при mount и при изменении `userId`.

## TypeScript: satisfies

### Вопрос

Когда `satisfies Config` уместнее, чем аннотация `: Config`?

### Проблема в ответе

Изначально ответа не было: тема была неизвестна.

### Правильная мысль

`: Config` задаёт переменной тип `Config`, из-за чего литеральные значения могут расширяться (`"dark"` → `string`).  
`satisfies Config` проверяет, что объект подходит под `Config`, но сохраняет точный выведенный тип значения: литералы, ключи и union-значения остаются узкими.

Используется для конфигов, роутов, тем и больших объектных структур, где нужна проверка формы без потери точных типов.

## TypeScript: discriminated union / narrowing по `status`

### Вопрос

Есть union `ApiResult` с вариантами `success | error | loading` и общим полем `status`. Как TS понимает вариант? Зачем `status` — discriminant? Чем `switch (result.status)` лучше `'data' in result`?

### Проблема в ответе

Изначально ответа не было. После разбора сформулировал верно, но тема не была известна до подсказки.

### Правильная мысль

TS **сужает тип** (narrowing) по значению общего поля с литеральным типом. После `if (result.status === 'success')` доступны только поля варианта `success`.

`status` — **discriminant**: по нему однозначно отличают варианты union.

`switch` по `status` опирается на **контракт API**; `'data' in result` проверяет только наличие поля и хуже масштабируется (легко пропустить `loading`, сломать при смене формы).

## JavaScript: `var` vs `let` в цикле + `setTimeout`

### Вопрос

```js
for (var i = 0; i < 3; i++) {
  setTimeout(() => console.log(i), 0);
}
```

Что выведется? Как получить `0, 1, 2` двумя способами? Чем `var` отличается от `let` в цикле?

### Проблема в ответе

Вывод `3, 3, 3` угадан верно, но сначала назван только способ с `let`. Объяснение через hoisting вместо **одной общей `i`** на весь цикл. Второй способ (IIFE/замыкание) не назван сразу.

### Правильная мысль

`var` — function scope, **одна** `i` на цикл; к моменту колбэков `i === 3`.

Исправления: (1) `let` — новая привязка на каждую итерацию; (2) IIFE/замыкание — сохранить `j` на каждый шаг для `var`.

## CSS: `position: sticky` vs `absolute`

### Вопрос

`.sticky { position: sticky; top: 0 }` внутри `.scroll { overflow: auto }`. Почему не работает? Чем sticky отличается от absolute? Как починить?

### Проблема в ответе

Сначала: «нужен `position: relative` у родителя» — это про **absolute**, не про sticky.

### Правильная мысль

`sticky` липнет относительно **scrollport / containing block**, не «позиционированного родителя». Частая поломка — `overflow: hidden` (или другой не-`visible` overflow) у предка: sticky заперт внутри контейнера. `position: relative` на родителе для sticky **не обязателен**.

## JavaScript: `this`, strict mode, `call` / `apply` / `bind`

### Вопрос

```js
const fn = obj.getName;
fn();
obj.getName();
```

Что выведется в strict и non-strict? Как гарантировать `'Shop'`?

### Проблема в ответе

Не различал strict/non-strict. Думал, `fn()` выведет `undefined`, а не **TypeError** в strict.

### Правильная мысль

`this` зависит от **способа вызова**. `obj.getName()` → `this = obj` → `'Shop'`. `fn()` без объекта: strict → `this === undefined` → ошибка; sloppy → `this` = global. Починить: `fn.call(obj)`, `apply`, `bind`, или не отрывать метод.

## Архитектура: FSD — слои

Что такое `shared`, `entities`, `features`, `widgets`? Может ли `features/cart` импортировать `features/checkout`?

### Проблема в ответе

`shared` описан неверно («импортирует shared-компоненты»). Слои entities/features/widgets — размыто.

### Правильная мысль

- **shared** — UI-kit, utils, api helpers **без бизнес-логики**.

## Дополнение: CORS (развёрнуто, сессия 4)

**Статус:** слабая тема → после разбора зачтено на добивку  
**Приоритет повторения:** высокий

### Что было не так в первом ответе

- CORS описан как «игнорирование запросов между площадками» — **неверно**.
  Правильно: браузер **по умолчанию блокирует чтение** cross-origin ответа; сервер **явно разрешает** доступ.
- Preflight (OPTIONS) приписан серверу — **неверно**.
  Preflight отправляет **браузер**, до основного запроса.

### Эталонный короткий ответ на собесе

1. **CORS** — механизм безопасности браузера: JS на `shop.example.com` не может читать ответ с `api.example.com`, пока API не разрешит.
2. **Preflight** — браузер шлёт `OPTIONS`, если запрос «не simple» (JSON body, `Authorization`, нестандартный method и т.д.).
3. **Заголовки сервера:**
   - всегда: `Access-Control-Allow-Origin`
   - при preflight: `Access-Control-Allow-Methods`, `Access-Control-Allow-Headers`
   - с cookies: `Access-Control-Allow-Credentials: true` + конкретный origin (не `*`)

### Запомнить дословно (добивка зачтена)

> Сервер разрешает CORS заголовками `Access-Control-Allow-Origin` (+ при preflight ещё `Allow-Methods` и `Allow-Headers`).

### Контрольные вопросы для повторения

- Почему `fetch` с фронта на другой домен падает без CORS-заголовков?
- Кто инициирует OPTIONS и в каких случаях?
- Чем отличается simple request от preflight?
- Почему с `credentials: 'include'` нельзя `Allow-Origin: *`?
- Где настраивать CORS: фронт или бэк? (ответ: **бэк**, фронт CORS не «чинит»)

### Слабые места (повторить)

#### CSS: специфичность

- Считать специфичность в формате `(id, class, element)`.
- При равной специфичности побеждает правило **ниже в файле**, а не «тег+класс всегда сильнее».
- `!important` работает только среди правил, которые **матчат элемент** (`header.title` не матчит `h1`).

**Контрольный пример:** `main .title` (0,2,1) vs `h1.title` (0,2,1) → побеждает тот, что ниже.

#### React: зависимости `useEffect`

- `[]` = только mount; смена prop/filter → нужен эффект с `[categoryId]` (или другим актуальным dep).
- Не класть в deps `products` / `setProducts` для перезагрузки по фильтру.
- Race condition: cleanup с `AbortController.abort()` или флаг `cancelled`.

#### HTTP: CORS

- Приоритет: **высокий**. Полный разбор — в блоке «CORS (развёрнуто, сессия 4)» выше.
- Запомнить: `Access-Control-Allow-Origin` (+ при preflight `Allow-Methods`, `Allow-Headers`).

#### JavaScript: Promise API

| API | Когда |
|-----|--------|
| `Promise.all` | все должны успеть; один reject → весь reject |
| `Promise.allSettled` | нужны статусы **всех** (успех + ошибка) |
| `Promise.race` | результат **первого** завершившегося |
| `Promise.any` | **первый успешный** |

#### Алгоритмы: оценка сложности

- Всегда называть **time + space**.
- Two Sum: time **O(n)**, space **O(n)** (храним до n элементов в `Map`).
- Почему лучше O(n²): один проход + O(1) lookup в map vs вложенный цикл.

### Эталонные короткие ответы (выучить)

**CSS:** «`h1.title` → orange; при равной специфичности с `main .title` побеждает нижнее правило; `!important` на `header.title` не меняет `h1`.»

**React:** «Пустой `[]` не реагирует на `categoryId`; deps = `[categoryId]`; в cleanup — `AbortController` для отмены старого fetch.»

**TS:** «Union по `ok`; сужаем через `if (response.ok)`; в `default` — `never` для exhaustiveness.»

**CORS:** «Сервер разрешает CORS заголовками `Access-Control-Allow-Origin` (+ при preflight `Allow-Methods` и `Allow-Headers`).»

**Promises:** «`allSettled` ждёт все промисы и не падает; упавший → `{ status: 'rejected', reason }`.»

**Two Sum:** «Один проход, map complement; O(n) time, O(n) space.»
