## Задача 1 — ✅ Сделана

**977. Squares of a Sorted Array** (Two Pointers, Easy)

### Что было
- Сначала пробовал квадратить весь массив и применять two pointers к несортированному результату — логика ломалась на центральном `0`.
- После наводящих вопросов перешёл на сравнение квадратов **исходного** отсортированного массива.

### Итоговое решение
- `while (start <= end)` — сравнение `array[start] ** 2` и `array[end] ** 2`
- `if / else` — за итерацию один элемент
- `reverse()` в конце для порядка по возрастанию

### Оценка
| Критерий        | Результат |
|-----------------|-----------|
| Корректность    | ✅        |
| Two Pointers    | ✅        |
| Edge cases      | ✅ (ноль, один элемент) |
| Сложность       | O(n) время, O(n) память |

**Уровень:** уверенный middle для Easy.

### Что улучшить
- Можно обойтись без `reverse()` — заполнять результат справа налево через индекс `k`.

## Задача 2 — ❌ Не доведена до конца

**496. Next Greater Element I** (Стек / brute force, Easy)

### Что было
- Условие понято по примерам.
- Выбран подход «в лоб»: два цикла — внешний по `nums1`, внутренний по `nums2`.
- Написан черновик `greatestNumbers`, но с багами:
  - не ограничен поиск **только справа** от найденного индекса;
  - может добавить **несколько** больших элементов вместо первого;
  - не добавляется `-1`, когда большего нет;
  - изначально `currentIndex > 0` ломало случай с элементом на индексе `0` (например, `1`).

### Ожидаемый выход
`nums1 = [4, 1, 2]`, `nums2 = [1, 3, 4, 2]` → `[-1, 3, -1]`

### Оценка
| Критерий        | Результат |
|-----------------|-----------|
| Понимание условия | ✅      |
| Рабочий код     | ❌        |
| Стек            | не применялся |

**Уровень:** идея верная, реализация не завершена (junior на этапе черновика).

### Что доработать
1. Найти индекс элемента в `nums2`.
2. Пройти от `index + 1` до конца — взять **первый** больший.
3. Если не нашли — записать `-1`.
4. (Опционально) решить за один проход по `nums2` со стеком — O(n).
